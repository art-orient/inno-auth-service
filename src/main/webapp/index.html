REST Best Practices Violation (Token in Body) Issue AuthController.java (Lines 69-72) 
@PostMapping("/validate")
public Long validate(@RequestBody String token) {
  return authUserService.validate(token);
}
Explanation: The access token is being passed in the request body. According to REST and HTTP standards, bearer tokens for validation or authorization should be passed in the Authorization header (e.g., Authorization: Bearer <token>).
DRY Violation (Token Generation) Issue JwtService.java (Lines 50-71) 
public String generateAccessToken(AuthUser user) { ... }

public String generateRefreshToken(AuthUser user) { ... }
Explanation: The methods for generating access and refresh tokens contain duplicated logic for building the claims map. This violates the DRY (Don't Repeat Yourself) principle and could be abstracted into a single private helper method.
Implicit Admin Creation Logic (SRP Violation) Issue AuthUserServiceImpl.java (Lines 39-41) 
boolean isFirstUser = userRepository.count() == 0;
Role role = isFirstUser ? Role.ADMIN : Role.USER;
user.setRole(role);
Explanation: The logic to assign the ADMIN role to the first registered user is hardcoded directly into the standard registration flow. This mixes system bootstrapping logic with standard business logic, violating the Single Responsibility Principle.
